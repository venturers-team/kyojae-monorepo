## 커스터마이징, 얕고 깊은 두 층

PaperClip에서 커스터마이징은 두 층위로 이루어집니다.

**얕은 수준**은 기존 gstack 구조는 그대로 두고, 한두 에이전트를 추가하거나 특정 에이전트의 AGENTS.md 프롬프트를 조정하는 작업입니다. 하루 이틀 안에 시도해 볼 수 있지요. 조직 전체를 재설계하지 않아도 됩니다.

**깊은 수준**은 완전히 다릅니다. 처음부터 자신의 문제에 맞춰 에이전트 구성·스킬 매니페스트·HEARTBEAT 스케줄을 새로 작성하는 작업이지요. 시간이 더 걸리지만, 이 과정을 한 번 거치면 PaperClip에 대한 이해가 근본적으로 달라집니다.

이 페이지에서는 얕은 수준 커스터마이징의 두 가지 길(에이전트 추가, 스킬 추가)을 안내해 드립니다. 깊은 수준은 이 스타터킷의 범위를 넘습니다. 후속 교재나 커뮤니티 레시피로 확장하시면 됩니다.

## 에이전트 한 명 고용하기

Org Chart 페이지 우측 상단에 **+ Hire** 버튼이 있습니다. 말 그대로 "채용" 버튼이지요. 클릭하면 새 에이전트 생성 다이얼로그가 열립니다.

| 필드 | 설명 |
|------|------|
| 이름 (Name) | 에이전트의 식별자. gstack과 겹치지 않게 지정 |
| 역할 (Title) | "Product Designer" 등 직책 |
| 보고 라인 (Reports To) | 상위 에이전트 지정. gstack에 추가한다면 보통 CTO |
| 어댑터 (Adapter) | 앞서 설정한 OpenRouter 어댑터 재사용 가능 |
| AGENTS.md 초안 | 직무기술서. 비어 있으면 기본 템플릿이 채워짐 |
| SOUL.md 초안 | 성격·가치관. 비어 있으면 기본값 |

고용이 완료되면 어떻게 될까요? Board Governance 규칙에 따라 여러분의 승인이 필요할 수 있습니다(Pending Approvals 카드에 항목이 뜹니다). 앞 장에서 배운 게 바로 여기서 쓰이는군요.

승인을 클릭하면 새 에이전트가 조직도에 추가됩니다. 기존 에이전트와 똑같이 Instructions·Skills·Configuration 탭을 갖게 되지요. 이 순간부터 새 에이전트는 CTO의 지시를 받아 Issue를 처리할 수 있습니다.

## 역할 정의가 전부입니다

에이전트 추가의 핵심은 딱 하나, **"어떤 역할을 맡을지 명확히 정의"하는 것**입니다.

예를 들어 여러분 회사에 "Technical Writer"라는 에이전트를 추가해 문서화를 전담시킨다고 해 봅시다. 기존 엔지니어들이 더 이상 문서 작성에 시간을 쓰지 않아도 되고, 문서 품질도 일관되게 유지되지요. 역할이 명확하니 효과가 분명히 나옵니다.

반대로 역할이 모호하면 어떻게 될까요? 기존 에이전트와 책임이 겹쳐서 오히려 혼란이 커집니다. "이거 내가 해? 네가 해?" 하는 상황이 반복되지요. AI도 사람과 똑같습니다. 역할이 불분명한 팀은 갈등이 생기지요.

## 스킬 한 개 추가하기

특정 에이전트에 새로운 "업무 매뉴얼"을 붙이고 싶을 때는 스킬을 추가합니다. 에이전트 상세 페이지의 **Skills** 탭 우측 상단에서 **+ Add Skill** 버튼을 누르면 되지요.

스킬은 마크다운 문서입니다. 다음 네 가지 요소를 담으면 됩니다.

1. **제목과 간단한 설명**: 에이전트가 어떤 상황에서 이 스킬을 호출할지 판단할 수 있도록
2. **입력 조건**: 이 스킬을 쓰려면 어떤 정보가 필요한가
3. **단계별 절차**: 스킬을 실행할 때 따를 순서
4. **완료 기준**: 무엇이 충족되면 스킬이 성공적으로 끝난 것으로 간주할지

말로만 설명하면 잘 와닿지 않지요. 구체적인 예시를 보여 드립니다.

Technical Writer 에이전트에 `publish-blog`라는 스킬을 추가한다고 해 봅시다.

- **제목**: "블로그 포스트 발행"
- **입력 조건**: "제목·대상 독자·핵심 메시지 세 가지가 주어진다"
- **절차**: "개요 작성 → 초안 작성 → 자기 리뷰 → Markdown 저장"
- **완료 기준**: "마크다운 파일이 `content/blog/` 아래에 저장되고 메타데이터가 채워진다"

이 스킬이 에이전트에게 붙으면, 향후 CEO가 "블로그 포스트가 필요하다"는 Initiative를 내릴 때 Technical Writer가 **자발적으로** 이 스킬을 호출합니다. 사람이 일일이 지시하지 않아도 되지요. 매뉴얼이 있는 만큼 자율 실행의 정확도가 올라갑니다.

## 끝으로 — 여정의 출발점

이 교재는 한 시간짜리 스타터킷입니다. 60분 안에 PaperClip의 기본 사이클을 한 바퀴 돌 수 있도록 설계됐지요. 하지만 이 60분이 여정의 끝이 아닙니다. 끝이 아니라 **출발점**이지요.

PaperClip은 플랫폼입니다. 플랫폼은 "올바른 사용법"을 강제하지 않습니다. gstack은 하나의 선택지이고, ClipHub의 다른 15개 템플릿도 각자의 선택지이지요. 여러분이 직접 설계할 조직도 또 다른 선택지입니다.

자기 앞의 문제를 관찰해 에이전트 팀을 구성하고, Budget·Governance로 운영의 안전장치를 마련하는 것이 이 도구의 핵심입니다.

## 언젠가 여러분의 템플릿이

gstack을 충분히 사용했다면, 다음에는 **자신만의 ClipHub 공개 패키지**를 만드는 것도 가능합니다. 여러분이 설계한 회사를 GitHub 저장소로 정리해 공개하면, 다른 학생과 개발자가 `npx companies.sh add <사용자명>/<저장소>`로 가져다 쓸 수 있지요.

언젠가 여러분의 템플릿이 누군가 후배의 스타터킷이 될지도 모릅니다. 그 전환이 이 교재가 바라는 진짜 끝입니다.

여기까지 함께 와 주셔서 정말 감사합니다. 이제 여러분의 AI 회사 이야기는 여러분이 직접 써 내려갈 차례입니다.
